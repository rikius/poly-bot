0) Clarify trading model: CLOB-first, not “risk-free”

Update Strategy doc language: “YES+NO < 1” is not risk-free on a CLOB unless you can reliably execute both legs at intended prices/sizes.

Add a section: “CLOB execution risks”
	•	Partial fills
	•	Price moves between legs
	•	Queue priority / maker-taker dynamics
	•	Tick size / min size
	•	Rate limits / websocket gaps
	•	Fee model & rounding

Define “risk-free” only at resolution if both legs acquired.

1) Promote “Order/Fills Ledger” to a first-class subsystem (before strategies)

Right now the plan focuses early on strategy/latency. Reorder phases so you build a truthful accounting layer first.

Add a strict internal ledger (authoritative state)

Implement these as distinct modules/data structures:
	•	OpenOrders (order_id → order metadata + status)
	•	Fills (trade_id → fill record)
	•	Positions (token_id → net shares, avg cost, realized pnl)
	•	Cash (USDC: available, reserved, total)
	•	PnL (realized + unrealized)

Rule: your bot must be able to reconstruct portfolio state from:
	1.	User WS trade events (primary)
	2.	REST reconciliation (secondary)

Add a “Reconciliation Loop”

Every N minutes (ex: 2–5 min):
	•	GET /orders?status=open (or equivalent)
	•	Cross-check with local OpenOrders
	•	Cancel stale / stuck / inconsistent orders
	•	Repair local state if needed
	•	Emit a “reconciliation report” metric/log

This should be in the plan before enabling live strategies.

2) Make execution a state machine (explicit, deterministic)

Add a dedicated section in Implementation Plan: “Execution Engine State Machine”

Order lifecycle states
	•	CREATED_LOCAL
	•	SIGNED
	•	SUBMITTED
	•	ACKED (got order_id)
	•	PARTIALLY_FILLED
	•	FILLED
	•	CANCEL_PENDING
	•	CANCELLED
	•	EXPIRED
	•	REJECTED
	•	UNKNOWN (requires reconciliation)

Invariants
	•	A fill must always map to an order (or be handled as “orphan fill” and reconciled)
	•	Reserved cash decreases only on ack OR on known fill policy (choose one, but be consistent)
	•	A position update happens only from fills, never from “expected fills”

3) Redesign “Math Arb” execution: treat it as 2-leg execution with hedging/unwind

Your current approach (concurrent join!) is good, but the plan needs the real execution logic.

Add “Pre-trade feasibility check”

Before sending orders:
	•	read top-of-book for YES and NO (ask levels if buying)
	•	compute max_size = min(size_yes_ask, size_no_ask)
	•	compute expected avg price with slippage estimate (even a crude one)
	•	require edge_after_fees > dynamic_min_edge

Add “Order type policy”

Define a default policy per strategy:
	•	Arb legs: prefer FOK if available, otherwise FAK
	•	Avoid leaving dangling exposure: do not use GTC unless it’s a deliberate maker strategy

Add “Partial fill handling” (mandatory)

If one leg partially fills and the other doesn’t:
	•	Option A (safer): immediate unwind filled leg (sell back at best bid) if combined edge is gone
	•	Option B: allow brief “hedge window” (e.g., 200–500ms) then unwind if not completed

This logic must be defined in the plan; otherwise arb will slowly turn into directional exposure.

4) Replace static thresholds with dynamic edge requirements

You have constants like:
	•	ARB_THRESHOLD = 0.97
	•	MIN_EDGE_CENTS = 0.03

Keep defaults, but update the plan to compute a dynamic minimum edge:

Add a function:

required_edge = fees + expected_slippage + partial_fill_risk + spread_penalty

Where:
	•	fees: maker/taker bps (from API)
	•	expected_slippage: based on depth & intended size
	•	partial_fill_risk: increases in thin books / volatile markets
	•	spread_penalty: avoid trading wide spreads unless edge huge

Decision rule becomes:
Execute only if
(1 - (yes_price + no_price)) >= required_edge

9) Change the implementation order (new phase ordering)

Rewrite your phases like this:

Phase 1 — Connectivity + Truth (must pass before trading)
	•	WS market feed ingestion + book build
	•	User WS fills/events ingestion
	•	Internal ledger + persistence snapshots
	•	Reconciliation loop

	•	small sizing caps
	•	strict risk limits

Phase 4 — Edge expansion
	•	temporal arb
	•	external spot feeds
	•	market selection scoring

Phase 5 — Optimization
	•	parsing, lock-free improvements, io_uring (optional)
	•	micro-latency tuning only after correctness + metrics
